# The Art of README

A README is the front door of your project. It should answer these questions in order:

1. **What is this?** (in one sentence)
2. **Why should I care?** (the problem it solves)
3. **How do I use it?** (the shortest path to value)
4. **How do I help?** (if OSS) or **How does it work?** (if internal)

Everything else is optional. A perfect README is one the reader can scan in 30 seconds and know whether this tool is for them.

## The Inverted Pyramid

Put the most important information first:

- Name + one-line description
- Installation (if short) or link to docs
- Usage example
- Everything else

## Know Your Audience

| Reader | Wants |
|--------|-------|
| Evaluator | "Should I use this?" |
| User | "How do I do X?" |
| Contributor | "How do I build/test this?" |
| Future you | "Why did I do it this way?" |

Write for the evaluator first. They have the least patience.
