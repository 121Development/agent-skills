# README Style Guide

## Common Mistakes

### The Wall of Text
❌ Paragraph after paragraph with no structure
✅ Use headers, lists, and code blocks. Scanability is kindness.

### The Mystery Meat
❌ "A tool for doing things"
✅ "Converts Markdown to PDF with automatic table of contents"

### The Tutorial Gap
❌ Only installation, no usage examples
✅ Show the command, then show the output

### The Stale Promise
❌ "Coming soon: docs"
✅ Don't mention it until it exists, or mark it clearly as WIP

### The Overexplainer
❌ Documenting every CLI flag in the README
✅ README = entry point. Full reference = separate docs.

## Prose Tips

- **Lead with the user.** "You can deploy..." not "This tool deploys..."
- **Use active voice.** "Run the command" not "The command should be run"
- **Show, don't tell.** Code examples beat descriptions
- **One idea per paragraph.** Dense text is unread text
- **Test your examples.** Copy-paste should work

## Formatting

- Code blocks need language tags: ```bash not ```
- Use relative links for internal references
- Keep line lengths reasonable (80-100 chars) for terminal reading
- Emoji are fine if they add signal, noise if overused
